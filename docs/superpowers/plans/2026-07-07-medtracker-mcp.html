MedTracker MCP Implementation Plan

For agentic workers: REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development, recommended, or superpowers:executing-plans to implement this plan task by task. Steps use checkbox syntax in the task details for tracking.

Goal
Add a secure, policy-aware Model Context Protocol server for MedTracker using modelcontextprotocol/ruby-sdk, starting with read-focused household context tools and explicitly deferring higher-risk writes.
Architecture
Mount the Ruby SDK StreamableHTTPTransport in Rails, authenticate with existing API app tokens, bind the same account, household, and membership context as Api::V1::BaseController, and expose MCP tools, resources, and prompts through service objects that reuse existing serializers, queries, Pundit scopes, tenant context, and audit events.
Tech Stack
Rails 8, Ruby, RSpec, Pundit, PaperTrail, SecurityAuditEvent, OpenTelemetry, modelcontextprotocol/ruby-sdk.

Product Decision

Why add an MCP?

MedTracker already has structured medication, schedule, adherence, inventory, health-event, and household-context data. MCP makes that data usable by AI clients without teaching every assistant the app's bespoke REST API, route structure, and portable export format.

The strongest v1 use case is assisted review: "What doses are due today?", "Which medication needs reordering?", "Summarise this person's recent medication history", and "Prepare a clinician handoff from my household data."

Advantages

  • Lower integration cost: one MCP contract works across MCP-capable clients instead of custom client glue per assistant.
  • Safer AI access: tools can be narrow, typed, audited, policy-scoped, and easier to reason about than free-form database or API access.
  • Better context quality: resources and prompts can provide domain-shaped context rather than raw JSON dumps.
  • Reuses existing boundaries: the app already has API tokens, tenant context, serializers, Pundit policies, portable snapshots, and audit logging.
  • Future product signal: Api::V1::CapabilitiesController already advertises client_tools.mcp_server as deferred, so this fills an existing product slot.

When not to build it

  • Do not build MCP if the goal is only mobile sync or CRUD integration; the existing REST API is the right surface for that.
  • Do not build it until authorization, tenant binding, audit logging, rate limiting, and token revocation are first-class in the MCP path.
  • Do not expose write tools until read tools have production telemetry and misuse patterns are understood.
  • Do not use MCP for emergency or clinical decision support; MCP responses must remain informational and user-mediated.
  • Do not expose raw portable exports as a default tool result; context should be minimized by task and person access.

Key Interfaces

  • Add gem "mcp", require: false or the exact gem name required by the current modelcontextprotocol/ruby-sdk release after confirming Bundler resolution.
  • Mount MCP::Server::Transports::StreamableHTTPTransport.new(server, stateless: true) at /mcp.
  • Create app/mcp/med_tracker_mcp/server.rb to assemble server metadata, tools, resources, prompts, and SDK configuration.
  • Create app/mcp/med_tracker_mcp/context.rb to authenticate bearer tokens with ApiSession.lookup_by_access_token and ApiAppToken.lookup_by_token, then bind TenantContext, Current.account, Current.household, and Current.membership.
  • Create read-only v1 tools: medtracker_current_user, medtracker_household_snapshot, medtracker_today_schedule, medtracker_inventory_risks, and medtracker_health_history_summary.
  • Create one prompt: medtracker_prepare_clinician_handoff, which asks the client model to summarise tool and resource output without inventing medical advice.
  • Update Api::V1::CapabilitiesController so client_tools.mcp_server.supported becomes true only after the route and auth tests pass.

Test Plan

  • Auth: missing, malformed, expired, revoked, locked-out, inactive membership, stale permissions version, and valid app-token cases.
  • Authorization: household manager sees full household-scoped data; carer, parent, and clinician users only see policy-granted people and medication records.
  • Tenant safety: cross-household IDs never leak; tools use Pundit scopes and tenant context rather than raw model lookups.
  • Tool behavior: output is bounded, deterministic, JSON-serializable, and stable for empty households, no medications, no takes today, and invalid date ranges.
  • Observability: each MCP request has request ID, tool name, actor membership, success/failure outcome, and no raw bearer token in logs or spans.
  • Security gates: task brakeman plus manual review of auth, authorization, token handling, rate limiting, audit trails, and health-data minimization.

Assumptions and Defaults

  • V1 is hosted Streamable HTTP MCP, not a local stdio-only helper, because MedTracker's value is authenticated household data.
  • V1 is read-only. Medication-taking and mutation tools are deferred until there is an explicit confirmation UX and stronger misuse telemetry.
  • Existing API app tokens are the initial credential type; no new OAuth provider is introduced in this plan.
  • All implementation commands use task; no direct bundle exec rspec or docker compose commands.
  • SDK details are based on current Context7 docs for /modelcontextprotocol/ruby-sdk: MCP::Server, class-based tools/resources, prompts, server_context, SDK configuration hooks, and Rails-mounted StreamableHTTPTransport.
